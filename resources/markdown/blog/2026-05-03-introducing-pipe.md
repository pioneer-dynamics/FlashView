## The Problem With Moving Data Between Machines

Every developer has been here: you need to get a database dump, a log file, an SSH key, or a quick snippet from one machine to another. The usual suspects — `scp`, `rsync`, Slack, email — all have friction. Either you need a shared key, a VPN, a file host account, or you're pasting sensitive data into a chat window and hoping nobody's watching.

We built **FlashView Pipe** to solve this cleanly.

## What Is FlashView Pipe?

`flashview pipe` is a CLI command that streams end-to-end encrypted data between two of your registered machines using nothing but your FlashView account. It's fully composable with Unix pipes — the sender reads from stdin, the receiver writes to stdout.

```bash
# On Machine A — send a compressed archive
tar cz ./my-project | flashview pipe --to DEVB4E1

# On Machine B — receive and decompress
flashview pipe | tar xz
```

That's it. No shared secrets to establish, no SSH config to fiddle with, no intermediate file host.

## How It Works

### Step 1 — Register Each Machine Once

Run `flashview pipe setup` on each machine. It generates a P-256 identity keypair locally and registers the public key with the FlashView server. The private key never leaves the machine.

```bash
flashview pipe setup
→ Device registered: DEVA3F2  (expires 2027-05-01)
→ This machine is now ready to receive pipe transfers.
```

### Step 2 — Send

Pipe anything to `flashview pipe --to <device-id>`:

```bash
# A database dump
pg_dump mydb | flashview pipe --to DEVB4E1

# A file
cat large-model-weights.bin | flashview pipe --to DEVB4E1

# Just text
echo "root password: $(cat /etc/shadow | head -1)" | flashview pipe --to DEVB4E1
```

The sender:
1. Generates a fresh 256-bit AES-GCM transfer key
2. Encrypts the payload with that key
3. Wraps the transfer key with the receiver's P-256 public key (ECIES: ECDH + HKDF + AES-GCM)
4. Attempts a direct P2P WebRTC connection to the receiver
5. Falls back to S3 relay if NAT traversal fails

### Step 3 — Receive

On the other machine, just run `flashview pipe`:

```bash
flashview pipe
Waiting for a transfer...
```

The receiver polls for an incoming session addressed to its device ID, decrypts the transfer key using its private key, and streams the plaintext to stdout.

## Two Delivery Modes

### P2P WebRTC (Direct)

When both machines are online simultaneously, Pipe attempts to negotiate a direct WebRTC data channel connection using STUN. If ICE negotiation succeeds, the encrypted payload travels machine-to-machine without touching any server storage — not even FlashView's S3. The server sees only the encrypted key blob and the WebRTC signalling messages.

```bash
echo "secret data" | flashview pipe --to DEVB4E1 --verbose
  Encrypting... [11 B]
  Uploading via p2p_webrtc... [43 B]
  [██████████████████████████████] 100.0% (43 B / 43 B)
Transfer complete.
```

P2P is abandoned only if ICE genuinely fails (e.g., both machines are behind symmetric NAT with no shared path). There is no arbitrary timeout — the transfer doesn't fall back just because one side started the command first.

### S3 Relay (Fallback)

If ICE negotiation fails, the sender uploads the already-encrypted payload to S3 and marks the session complete. The receiver can then retrieve it at any time — even hours later. This path works across any network and doesn't require both machines to be online at the same time.

```bash
echo "secret data" | flashview pipe --to DEVB4E1 --verbose
  Encrypting... [11 B]
  Uploading via s3_direct... [43 B]
  [██████████████████████████████] 100.0% (43 B / 43 B)
Transfer ready. Run 'flashview pipe' on the receiving machine.
```

In both modes the payload is encrypted before it leaves the sender — the server never holds plaintext.

## Security Design

### End-to-End Encryption

Every transfer uses a fresh random 256-bit AES-GCM key generated by the sender. That key is encrypted with the receiver's P-256 public key via ECIES (ECDH key agreement → HKDF → AES-GCM key wrap). Only the receiver — with its private key stored locally — can unwrap it.

The server stores:
- The encrypted key blob (useless without the receiver's private key)
- The encrypted payload (useless without the transfer key)
- The WebRTC signalling messages (only the SDP offer/answer and ICE candidates — no payload)

### Forward Secrecy

Each transfer uses a fresh key, used once and discarded. Compromising one transfer key reveals nothing about past or future transfers. There is no shared long-term secret between sender and receiver that could be harvested.

### What the Server Never Sees

- Plaintext data
- The AES-256-GCM transfer key
- Either machine's P-256 private key

Even if FlashView's infrastructure were completely compromised, an attacker would only obtain encrypted ciphertext and an encrypted key blob — both worthless without the receiver's local private key.

## Composability

Because Pipe reads from stdin and writes to stdout, it fits naturally into any Unix pipeline:

```bash
# Sync a directory
tar cz ./src | flashview pipe --to DEVB4E1
flashview pipe | tar xz -C ./src

# Transfer a database
pg_dump mydb | gzip | flashview pipe --to DEVB4E1
flashview pipe | gunzip | psql mydb

# Transfer stdout from a long-running command
./run-benchmark.sh | flashview pipe --to DEVB4E1

# Redirect to a file
flashview pipe > backup.tar.gz
```

1. Install the CLI: `npm install -g @pioneer-dynamics/flashview-cli`
2. Login: `flashview login`
3. Register each machine: `flashview pipe setup`
4. Send: `echo "hello" | flashview pipe`
5. Receive: `flashview pipe`
